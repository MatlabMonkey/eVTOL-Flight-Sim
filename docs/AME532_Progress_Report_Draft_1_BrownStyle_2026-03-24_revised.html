1. Project Scope Update

The original proposal had three main goals: build a 6-DoF simulation of a modern VTOL aircraft, create scenarios for analyzing the hover-to-cruise transition, and create a framework for comparing different control methods in the transition region. During this reporting period, the project became much more focused. Instead of using a generic VTOL concept with placeholder properties, the model is now based on an Archer Midnight-like lift+cruise configuration with updated geometry, mass properties, aerodynamic coefficients, rotor layout, and control surfaces. The comparison framework is still under development, but the trim workflow, transition metrics, and controller scaffold are now in place.

Original project specification Current update
Produce an accurate 6-DoF VTOL simulation using a contemporary aircraft design. A running Simulink model now exists for a lift+cruise eVTOL based on an Archer Midnight-like configuration, with updated aircraft geometry, mass properties, and aerodynamic coefficients.
Create scenarios that allow the transition region to be analyzed and evaluated. Scenario initialization has been separated into scenario_def.m, and the validation suite now includes aircraft-scaled cruise and force-response cases.
Create a framework for comparing transition-control methods or configurations. A transition-metrics script, trim workflow, and simple PD controller scaffold are now integrated, although the controller is still untuned.

2. Aircraft Model and Property Update

The largest milestone in this period was replacing the original placeholder aircraft setup with a more defensible aircraft definition. The current model includes a fuselage, two half-wings, a V-tail, six front tilting rotors, and six rear fixed rotors. The initialization workflow was also cleaned up by separating the aircraft definition from the scenario definition, which makes the model easier to modify and test. In parallel, the aerodynamic and mass properties were updated using NASA lift+cruise reference-vehicle data as a much better first-pass basis than the original hobby-aircraft-style values. The main reference used for that update was NASA/TM-20210017971.

3. Control Surfaces and Model Updates

The aircraft now includes simplified aerodynamic control surfaces: one ruddervator on each V-tail half and one flaperon on each wing half. This is a deliberate simplification of the likely segmented Archer control layout, but it keeps the model manageable while still giving the simulation realistic aerodynamic control effectors in forward flight. The rendered model was used during development to verify hinge placement, sign convention, rotor tilt geometry, and control-surface orientation. In other words, the visualization is useful because it shows the geometry that was added to the model, not because visualization itself is a project milestone.

The aerodynamic surface blocks were also updated to accept local control-deflection inputs, so the model now has a direct path from control commands to aerodynamic force and moment changes on each wing half and each V-tail half. The first-pass ruddervator estimates currently used in the repo give approximately CMδe = -0.487 / rad and CNδr = +0.059 / rad.

4. Controller, Trim, and Validation Progress

A simple PD controller scaffold has now been integrated into Brown_Full_Sim.slx. The current block reads Euler angles and body rates and produces local flaperon commands, local ruddervator commands, and a motor RPM vector. This should be described as control scaffolding rather than a finished control law. The purpose of this update was to create the closed-loop path in the model and verify that the controller block can execute. It does that successfully, but it is not yet tuned to hold a trimmed condition or perform a controlled transition.

The trim workflow also improved during this period. The trim helper now runs from the repo correctly and returns first-pass actuator estimates. For a 70 m/s cruise target with 90 deg front-rotor tilt, the current fallback estimate is 1063.6 rpm for the front rotors to cover cruise drag and 6435.8 rpm for the rear rotors to support full weight. These values are useful starting points for later operating-point work, but they are not yet a fully converged 6-DoF trim solution.

The validation suite was updated so that the case values now make sense for the current aircraft instead of older placeholder conditions. The main smoke test passes, the full Simulink model compiles and runs, and the updated tester cases now execute at plausible cruise-like conditions. That means the plant model, aircraft definition, and validation workflow are now aligned much better than they were in the original proposal state.

5. Transition Metrics and Next Steps

The most important progress on the evaluation side was deciding on the transition metrics that should be used going forward. The two metrics selected for the current stage are transition_time_s and jerk_rms. Transition time was chosen because it is the clearest mission-level measure of whether the maneuver is practical. Jerk RMS was chosen because it captures how aggressive or uncomfortable the maneuver is and gives a useful smoothness measure without requiring a fully developed electrical power model. Other quantities such as energy proxy and control smoothness are still useful, but they are secondary at this stage because propulsion scaling and actuator modeling are still being refined.

The next control step is not to replace this with a large monolithic controller, but to tune the current inner-loop PD scaffold around realistic operating points and then use the trim estimates as initial actuator guesses. After that, the transition cases can be run with the selected metrics and compared in a consistent way. So the most accurate summary of the current project state is that the plant model, aircraft definition, and evaluation framework have advanced substantially, and the project is now ready for controller tuning and transition-performance work, but it is not yet at the stage of a successful controlled transition.

Plant Instability Investigation

Date: 2026-04-08

Goal

Find the reason the 70 m/s cruise seed was drifting strongly in pitch and why direct trim() on the full plant kept converging to nonphysical operating points.

Bottom Line

  1. Confirmed bug: the propeller group blocks were reading row-wise position matrices as if each motor position lived in a column. This corrupted the propeller moment arms and produced the wrong pitch/yaw moments.
  2. Confirmed tooling bug: the programmatic model builder was deleting the Demux blocks and wiring the full 12x1 RPM vector into all four prop groups, which made every group read the same first three RPM entries.
  3. Remaining issue: after fixing the propeller bugs, the plant still shows a smaller residual pitch-moment mismatch near the 70 m/s seed, and direct raw trim() on the full plant still wanders to bad branches.

Confirmed Root Cause #1: Prop Position Matrix Orientation

The prop location arrays in aircraft_def.m are stored row-wise:

prop.posFR =
    [1.95  1.80 -0.45
     1.95  4.20 -0.45
     1.95  6.50 -0.45]

But the library block code originally used:

r_hub = pivot_pos(:, i) + hub_offset * n;

That line treats each motor location as a column. For the matrix above, the first “motor” becomes [1.95; 1.95; 1.95], which is wrong.

Evidence

The simulated front-group moments before the fix were:

FR moment = [  -20.937  1444.933  -954.708] N-m
FL moment = [   20.937  -732.472   452.230] N-m

Using the same equations but feeding the front prop positions without transposing reproduces those moments almost exactly:

Using untransposed posFR
  Sum F=[502.4797 0.0000 0.0000] M=[-20.9367 1444.9376 -954.7114]
Using untransposed posFL
  Sum F=[502.4797 0.0000 0.0000] M=[20.9367 -732.4743 452.2317]

This is a confirmed bug, not just a hypothesis.

Fix Applied

The library was patched so the prop blocks read rows instead of columns:

pivot_pos(i, :).'
prop_pos(i, :).'

Patch helper:

scripts/simulink/patch_propeller_library_indexing.m

Confirmed Root Cause #2: Builder Reintroduced Bad RPM Routing

The model-builder helper had a function that explicitly deleted the Demux blocks and connected the full 12x1 motor RPM vector into all four prop groups:

add_line(propPath, phRPM.Outport(1), phFR.Inport(1), ...)
add_line(propPath, phRPM.Outport(1), phFL.Inport(1), ...)
add_line(propPath, phRPM.Outport(1), phRR.Inport(1), ...)
add_line(propPath, phRPM.Outport(1), phRL.Inport(1), ...)

That meant all four groups read the same rpms(1:3) entries. This bug matched the earlier symptom where front collective thrust also showed up as a rear-style downward force.

The builder was patched to keep explicit Demux blocks:

  • RPM Demux: [3 3 3 3]
  • Tilt Demux: [3 3]

What Improved After the Fixes

After patching the library, the propeller sums now match the analytical model exactly at the 70 m/s seed:

QuantitySimAnalyticalDifference
Prop Force [1004.955889, 0, 0] [1004.955889, 0, 0] 0
Prop Moment [0, -242.247262, 0] [0, -242.247262, 0] 0

Under a pitch-rate perturbation, the plant and analytical model now agree reasonably well on the sign and magnitude of the aerodynamic pitch moment:

q0      sim My_aero    ana My_aero
-0.100   774.28         692.41
 0.000   302.88         242.25
 0.100  -168.10        -207.48

This suggests pitch-rate damping is not wildly wrong.

What Is Still Not Fixed

The 70 m/s seed is still not an exact equilibrium in the EUL plant. The saved state after 2 seconds still drifts noticeably:

Euler final [phi theta psi] = [0.0301 0.5508 0.0023] rad
Omega final [p q r]         = [0.0172 0.2492 -0.0057] rad/s
V_body final [u v w]        = [74.1386 0.0198 2.2225] m/s

The total seed force/moment comparison still shows a smaller residual mismatch in the aerodynamic part:

Aero Sum Moment
  sim : [-2.470221 302.878557 -0.420437]
  ana : [ 0.000000 242.247262  0.000000]
  diff: [-2.470221  60.631296 -0.420437]

That means the propeller implementation was a major issue, but not the only issue.

Current Best Hypothesis for the Remaining Instability

  1. Residual aerodynamic pitch-moment mismatch: the plant still has about +60.6 N-m more nose-up aerodynamic moment than the analytical residual model at the 70 m/s seed.
  2. Raw full-plant trim formulation is still poor: even after the prop fixes, direct trim() on the full plant remains underconstrained and unbounded enough to escape to nonphysical branches.
  3. Possible truth/output debugging mismatch: some top-level outputs appear less reliable than the saved state dataset, so state-output interpretation should be done from the saved state or directly from subsystem logs.

Recommended Next Checks

  1. Compare left/right wing and tail moments individually under small alpha perturbations and small q perturbations to isolate where the remaining +60.6 N-m pitch mismatch comes from.
  2. Use the corrected plant with the helper/manual trim workflow to regenerate the 70 m/s operating point, then verify the corrected seed in the full EUL or Zach plant.
  3. Only after the seed is verified in the corrected plant, retry direct full-plant trim().
